# M2.1 — Schema and Basic Stats Profiler

## Context

M2.1 is the first real code milestone. It establishes the Python module that becomes the foundation for the Dataset Analyzer agent later in M2.3. The user asked a brainstorm question: how do we bridge deterministic Python with agents/skills, and what tooling setup (Makefiles, requirements.txt, pyproject.toml, etc.) do we commit to?

This plan resolves those decisions, defines what M2.1 actually delivers, and sets the patterns that every later milestone will follow.

---

## Key Design Decisions

### 1. Python Tooling: requirements.txt + pip (no Makefile, no pyproject.toml)

| Option | Pros | Cons |
|--------|------|------|
| **requirements.txt + pip** (recommended) | PRD-mandated; simple; universal | Slower installs, no lock file |
| pyproject.toml + uv | Modern, fast, lock file, PEP 517 | Not in PRD; adds tooling complexity |
| Makefile | Wraps common commands | Overkill at this stage; add at M5 if needed |

**Decision:** Follow the PRD (`requirements.txt`, pip). Revisit `uv` upgrade at M5 if install speed becomes friction.

---

### 2. Agent–Python Bridge

The architecture is: **Agents orchestrate; Python executes deterministically.**

The bridge is a `Bash` tool call from a Claude agent:

```bash
.venv/bin/python -m src.analysis.profiler \
  --input  projects/titanic/data/raw/train.csv \
  --output projects/titanic/artifacts/data/profile.json \
  --project projects/titanic/project.yaml
```

**Bridge contract (baked into profiler.py from M2.1):**
- Explicit interpreter path (`.venv/bin/python`, never bare `python`)
- Non-zero exit code on failure
- On failure: writes `{"error": true, "message": "...", "traceback": "..."}` to output path
- Idempotent: same input → identical JSON every time (deterministic ordering, stable SHA-256)
- Self-creating output directory (`mkdir parents=True, exist_ok=True`)
- All paths come from CLI args — no path discovery inside the module

The **Dataset Analyzer Claude agent** is NOT built in M2.1. It wraps these Python modules in M2.3 once the modules are stable and tested.

### 4. profile.json — Partial Schema (M2.1 scope only)

M2.1 writes a **partial profile**. M2.2/M2.3 extend it. `m2_sections_pending` prevents agents from using an incomplete profile.

```json
{
  "profiler_version": "0.1.0",
  "generated_at": "<ISO 8601 timestamp>",
  "source": {
    "path": "projects/titanic/data/raw/train.csv",
    "sha256": "<hash>",
    "row_count": 891,
    "column_count": 12,
    "file_size_bytes": 61194
  },
  "schema": {
    "columns": [
      {
        "name": "PassengerId",
        "pandas_dtype": "int64",
        "inferred_semantic_type": "identifier",
        "sample_values": [1, 2, 3]
      }
    ]
  },
  "basic_stats": {
    "Age": {
      "count": 714, "mean": 29.70, "std": 14.53,
      "min": 0.42, "p25": 20.12, "p50": 28.0, "p75": 38.0, "max": 80.0
    },
    "Sex": {
      "count": 891, "top": "male", "freq": 577, "unique": 2
    }
  },
  "m2_sections_complete": ["schema", "basic_stats"],
  "m2_sections_pending": ["null_analysis", "cardinality", "outliers", "correlation", "target_validation", "leakage_flags"]
}
```

**Key design choices:**
- `inferred_semantic_type` per column — starts the semantic layer early (heuristics: all-unique int → `identifier`, 2-value int → `binary_target_candidate`, low-cardinality object → `low_cardinality_categorical`, float → `continuous_numeric`)
- Named percentile keys (`p25`, `p50`, `p75`) — explicit, no pandas abbreviations
- SHA-256 of source — satisfies PRD reproducibility requirement
- **No `NaN` in JSON** — custom encoder, `allow_nan=False`; float NaN is not JSON-spec-compliant and breaks downstream parsers

## Test Approach

```
tests/
├── __init__.py
├── fixtures/
│   ├── simple_numeric.csv      (3 numeric cols, no nulls, 20 rows)
│   ├── mixed_types.csv         (numeric + categorical, 20 rows)
│   └── all_string.csv          (edge case: all object dtype)
└── analysis/
    ├── __init__.py
    └── test_profiler.py
```

Key test cases:
- Schema infers all 12 Titanic column dtypes correctly
- Semantic type heuristics (`identifier`, `binary_target_candidate`, `continuous_numeric`, `low_cardinality_categorical`)
- Numeric stats keys complete (`count`, `mean`, `std`, `min`, `p25`, `p50`, `p75`, `max`)
- Categorical stats keys complete (`count`, `top`, `freq`, `unique`)
- SHA-256 stable across two runs on same file
- Bad path → exit code 1, error JSON written
- No `NaN` in output JSON (`json.loads()` succeeds)
- Fixture files (simple_numeric, mixed_types) run cleanly

Run: `.venv/bin/python -m pytest tests/analysis/test_profiler.py -v`
